# ADR 0001: Java 21, Spring Boot, gRPC

## Статус

Принято.

## Контекст

По челленджу нужен осознанный стек, DDD/CQRS, и не хочется «просто REST CRUD из гайда».

## Решение

- **Java 21** и **Spring Boot 3.3** — нормальная база под JPA, миграции, метрики, тесты.
- **gRPC** как основной контракт (protobuf, явная схема, `auth.v1`).
- **PostgreSQL** + **Flyway** для схемы.
- **JWT** на короткий access, **opaque refresh** в БД с ротацией и проверкой reuse по семейству токенов.

## Что отвергли

| Вариант | Почему нет |
|---------|------------|
| Kotlin + Ktor | Меньше готовых кирпичей под JPA за те же сроки. |
| GraphQL | Для набора auth-команд избыточно; gRPC ближе к RPC. |
| Только Keycloak | Нет собственной доменной проработки — мимо задания. |
| Только REST | По критериям явный минус без сильной аргументации. |

## Последствия

- Клиентам нужен gRPC или gateway (Envoy, grpc-gateway и т.п.) для JSON/HTTP.
- Версии API — через суффикс пакета в proto (`v1`, потом `v2`).
